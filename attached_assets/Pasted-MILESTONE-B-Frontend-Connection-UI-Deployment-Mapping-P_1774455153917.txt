Behaviour:

The "OpenAI Model" dropdown should list known Azure-available models: gpt-4o, gpt-4o-mini, gpt-4.1, gpt-4.1-mini, gpt-4.1-nano, o4-mini, o3, o3-mini, o1, o1-mini. When the admin selects a model, auto-fill the pricing columns with the default prices for that model.
The admin can override prices (for enterprise Azure contracts with custom pricing).
Base URL validation: On input, validate that the URL starts with https:// and matches one of the known Azure hostname patterns (*.openai.azure.com or *.services.ai.azure.com). Strip any trailing slashes. Show a warning (not an error) if the hostname doesn't match either pattern — the admin might have a custom endpoint or proxy in front.
Endpoint mode defaults to v1. When Legacy is selected, show the API Version field. When v1 is selected, hide it.
"Validate & Save" calls the backend which runs validateAzureOpenAIKey() against the first deployment using the selected endpoint mode, encrypts the API key, and stores everything.
At least one deployment mapping is required.
Deployment names must be unique within the connection and cannot conflict with model names used by other providers (i.e. a deployment cannot be named claude-sonnet-4-20250514 or gemini-2.5-flash as those would match prefix routing for other providers).

B3. Azure OpenAI Connection API Routes
Add these API routes (follow the pattern of existing provider connection routes):
POST   /api/providers/azure-openai/connect
  Body: { displayName, apiKey, baseUrl, endpointMode, apiVersion?, deployments: [...] }
  Action: Validate key → encrypt → store ProviderConnection with provider=AZURE_OPENAI
  
PUT    /api/providers/azure-openai/update
  Body: { displayName?, apiKey?, baseUrl?, endpointMode?, apiVersion?, deployments?: [...] }
  Action: Re-validate if key/baseUrl/endpointMode changed → update connection
  Note: When deployments change, invalidate the Redis cache key allotly:azure-deployments:{orgId}

DELETE /api/providers/azure-openai/disconnect
  Action: Remove connection → cascade: invalidate Redis cache, remove from member allowedProviders
The connect route must:

Validate input (Zod schema — require baseUrl starts with https://, endpointMode is "v1" or "legacy", apiVersion required if legacy mode, deployments array non-empty)
Normalize baseUrl (strip trailing slashes, strip trailing /openai if present)
Check that deployment names don't collide with other providers' prefix patterns
Call validateAzureOpenAIKey() with the selected endpoint mode
Encrypt API key with AES-256-GCM
Store ProviderConnection with all Azure fields
Invalidate Redis deployment cache
Return success with the connection ID

IMPORTANT ARCHITECTURAL NOTES
Why deployment names live in ProviderConnection, not in the model catalog
The model catalog is org-agnostic — it defines standard models and pricing. But Azure deployments are per-org (each customer has their own Azure resource with their own deployment names). So the mapping from deployment name → underlying model → pricing lives in the ProviderConnection.azureDeployments JSON field, not in the global catalog.
Conflict resolution: What if a deployment name matches a prefix?
If a customer names their Azure deployment gpt-4o-custom, the prefix routing (gpt-* → OPENAI) would catch it before the Azure deployment lookup runs. This is intentional — prefix routing for the three core providers takes priority. The deployment name collision check in B3 should warn admins if their deployment name starts with gpt-, claude-, gemini-, o1, o3, or o4, but not hard-block it. The admin might intentionally want to override the direct OpenAI route with their Azure-hosted version. In that case, document that they should use a non-conflicting name.
ACTUALLY — on reflection, hard-block it. If the admin names their Azure deployment gpt-4o, requests for gpt-4o will route to direct OpenAI, not Azure, and the admin will be confused. Require deployment names that don't match any existing provider prefix pattern. Show a clear validation error:

"Deployment name 'gpt-4o' conflicts with OpenAI's model naming convention. Azure deployment names must not start with 'gpt-', 'o1', 'o3', 'o4', 'claude-', or 'gemini-'. Use a custom name like 'my-gpt4o' or 'nebula-one'."

Provider priority in detectProvider()
The routing order is:

Prefix match → OPENAI, ANTHROPIC, or GOOGLE (fast, no IO)
Azure deployment lookup → AZURE_OPENAI (Redis cache hit, or DB fallback)
No match → error

This ensures existing provider routing is never disrupted by Azure.
Budget accuracy for Azure
Because the admin provides per-deployment pricing in the connection config, and Allotly uses those prices for budget estimation and adjustment, pricing accuracy depends on the admin entering correct values. The auto-fill from the OpenAI model dropdown gives good defaults, but enterprise Azure contracts may have different rates. The UI should make it clear that pricing accuracy depends on these values.
