The Most Likely Causes (in order of probability)
Cause 1: Wrong header name
The header MUST be exactly api-key (lowercase, with hyphen). Not Api-Key, not API-KEY, not apiKey, not x-api-key, not Authorization.
javascript// WRONG — any of these will get 401
headers['Api-Key'] = key;
headers['API-KEY'] = key;
headers['apiKey'] = key;
headers['x-api-key'] = key;
headers['Authorization'] = 'Bearer ' + key;

// CORRECT — exactly this
headers['api-key'] = key;
Note: HTTP headers are technically case-insensitive per the spec, but Azure APIM gateways are known to be case-sensitive in some configurations. Use lowercase api-key to be safe.
Cause 2: User's Authorization header is being forwarded
When the user sends Authorization: Bearer allotly_sk_... to Allotly, that header must be REMOVED before calling Azure. If it's forwarded alongside api-key, Azure APIM may reject the request because:

It sees a Bearer token it doesn't recognize
APIM prioritizes Authorization over api-key

The fix:
javascript// Build a CLEAN headers object for Azure — don't copy from the user's request
const azureHeaders = {
  'Content-Type': 'application/json',
  'api-key': azureConfig.apiKey
};
// That's it. No other headers. Don't spread ...userHeaders or anything like that.
Cause 3: URL is wrong
The working URL is:
https://apim-n1ai-uks-f31ee48d7.azure-api.net/openai/deployments/gpt-4.1-nano/chat/completions?api-version=2024-10-21
Common URL bugs:

Double slash: https://...net//openai/... (if baseUrl has trailing slash)
Missing api-version: https://...net/openai/deployments/gpt-4.1-nano/chat/completions (no query param)
Wrong path: https://...net/v1/chat/completions (OpenAI path, not Azure path)
Deployment name not stripped: https://...net/openai/deployments/azure/gpt-4.1-nano/... (didn't strip azure/ prefix)
Encoded slashes: https://...net/openai/deployments/azure%2Fgpt-4.1-nano/...

Build the URL explicitly:
javascriptfunction buildAzureUrl(baseUrl, deploymentName, apiVersion) {
  // Remove trailing slash from baseUrl
  const base = baseUrl.replace(/\/+$/, '');
  // deploymentName should NOT have the azure/ prefix — strip it before calling this
  return `${base}/openai/deployments/${deploymentName}/chat/completions?api-version=${apiVersion}`;
}
Cause 4: API key was corrupted in storage
The raw key is: 456eccb2a4cb43e1a35c7d2f4a86b1da (32 characters, hex only)
If the key was:

URL-encoded: 456eccb2a4cb43e1a35c7d2f4a86b1da → shouldn't change, but check
Base64-encoded: would be completely different
Encrypted at rest and not decrypted before use
Trimmed/truncated
Stored with surrounding quotes: "456eccb2..." (extra quote characters)
Stored with a prefix: Bearer 456eccb2...

Log the key length and first 8 chars to verify: console.log(apiKey.length, apiKey.substring(0,8)) → should be 32, "456eccb2"
Cause 5: Extra headers confusing APIM
APIM gateways can be picky about unexpected headers. If Allotly is forwarding headers like X-Forwarded-For, X-Request-ID, cookies, or other proxy headers, APIM might reject the request.
Send ONLY these headers to Azure:
javascript{
  'Content-Type': 'application/json',
  'api-key': azureConfig.apiKey
}
Nothing else. No trace headers, no forwarded headers, no custom headers.
