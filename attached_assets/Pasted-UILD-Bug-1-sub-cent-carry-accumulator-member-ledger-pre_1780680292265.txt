UILD — Bug 1: sub-cent carry accumulator (member-ledger precision). CRITICAL money path — extra care, verify before you change anything.

Invariant that must hold: a member can never be billed less than their true cumulative spend, the cap must always eventually trip, and every displayed number stays whole-cent (no micro-cent migration — that was rolled back on purpose; do NOT reintroduce it).

What & why. Settlement does Math.round(weighted / 1_000_000), so any request truly costing < 0.5¢ settles to 0¢ and the member ledger debits nothing — an unbounded, compounding under-count (10k × 0.3¢ ≈ $30 recorded as $0; the cap never trips). Fix: accumulate the true sub-cent cost in a hidden remainder and debit whole cents only when it crosses 1¢. The unit of every visible field stays whole cents.

Prerequisite/compat: this builds on the Bug 2 reservation change (already submitted). Bug 1 touches only settlement + the member ledger; Bug 2 touched only reservation. They don't collide — build on top.

DESIGN — the canonical helper.
Add calculateSettledCostMicroCents(usage, pricing): number in safeguards.ts — same math as calculateSettledCostCents but returns the integer micro-cent value (Math.round(weightedCentsPerMTok)), without the /1_000_000 + round-to-cents. Keep calculateSettledCostCents as-is for any non-ledger caller.

Add settleSpendWithCarry(membershipId, costMicroCents): Promise<{ crossedCents: number; newSpendCents: number }> running ONE atomic statement:

WITH prev AS (
  SELECT cost_remainder_micro_cents AS old_rem
  FROM team_memberships WHERE id = $1 FOR UPDATE
), upd AS (
  UPDATE team_memberships m
  SET cost_remainder_micro_cents = (prev.old_rem + $2) % 1000000,
      current_period_spend_cents = m.current_period_spend_cents
                                 + (prev.old_rem + $2) / 1000000
  FROM prev WHERE m.id = $1
  RETURNING m.current_period_spend_cents AS new_spend
)
SELECT ((prev.old_rem + $2) / 1000000)::bigint AS crossed_cents, upd.new_spend
FROM prev, upd;
$2 = costMicroCents bound as bigint. RETURNING derives crossed_cents from the OLD remainder via the CTE (a plain UPDATE…RETURNING only sees post-update values — don't use that). old_rem + $2 is bigint so no INT overflow; only <1e6 is stored.
