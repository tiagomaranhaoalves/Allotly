Replit Prompt — Fix estimate_cost alternatives: cent-rounding collapses cheaper models
What & Why
The estimate_cost MCP tool (server/lib/mcp/tools/consumption/estimate-cost.ts) returns a correct max-cost estimate, but its cheaper-alternatives array is wrong at low max_tokens. Live evidence against prod (synthetic voucher key, "hello world" prompt, max_tokens=500):

Requested model	Reported max cost	Alternatives
claude-opus-4-1	$0.05	3 (gpt-4o, gpt-4o-mini, gpt-4-0613)
gpt-4o	$0.02	0 ← wrong: gpt-4o-mini is ~16× cheaper and in the allowlist
claude-sonnet-4-5	$0.02	0 ← wrong: Haiku / mini / flash are cheaper
Confirmed root cause (verify before changing anything)
maxCostCents rounds each component independently with Math.ceil to whole USD cents:

function maxCostCents(inputTokens, maxOutputTokens, pricing) {
  const inputCost  = Math.ceil((inputTokens   * pricing.inputPricePerMTok)  / 1_000_000);
  const outputCost = Math.ceil((maxOutputTokens * pricing.outputPricePerMTok) / 1_000_000);
  return inputCost + outputCost;
}
At small token counts, every model's two components each round up to a minimum of 1 cent, so almost every model evaluates to exactly 2 cents. The alternatives filter then does x.cost < requestedCost. When the requested model is also 2 cents, the strict < excludes every same-floor model — even ones that are genuinely 10–100× cheaper in true price. The bug disappears at large max_tokens (e.g. the 4096 default) because the per-component costs separate above the 1-cent floor, which is why nobody caught it until a low-max_tokens preview.

This is the only confirmed defect. Do not assume there are others. In particular, the "alternatives were all OpenAI for the opus case" observation is probably correct behaviour — alternatives are filtered to filteredProviders (active provider connections ∩ the membership's allowedProviders), so an org with only OpenAI connections active will only ever see OpenAI alternatives. Investigate and confirm whether that is intended; do not change it unless you find it's actually wrong.

Steps
Reproduce and confirm the root cause with a unit test before fixing: assert that today, two models with very different true prices but both under the 1-cent-per-component floor produce equal maxCostCents, and that the cheaper one is wrongly excluded from alternatives. This codifies the bug.
Add a precise (unrounded) cost function alongside the existing maxCostCents, e.g.:
// True fractional-cent cost for ranking/comparison. Do NOT round per component.
function preciseCostCents(inputTokens: number, maxOutputTokens: number, pricing: ModelPricing): number {
  return (inputTokens * pricing.inputPricePerMTok + maxOutputTokens * pricing.outputPricePerMTok) / 1_000_000;
}
Keep maxCostCents (the conservative whole-cent ceil) for display of both the requested model and each alternative.
Rewrite the alternatives pipeline to rank/filter on preciseCostCents:
Compute requestedPrecise = preciseCostCents(...) for the requested model.
For each candidate (after the existing provider/allowlist/vision filters, unchanged): compute candidatePrecise.
Keep candidates where candidatePrecise < requestedPrecise (strict, on the unrounded value).
Sort ascending by candidatePrecise.
slice(0, MAX_ALTERNATIVES).
For each surviving alternative, compute its display cost with the existing maxCostCents (ceil) and buildAmountDisplay, and compute savings_pct from the precise values: Math.round(((requestedPrecise - candidatePrecise) / requestedPrecise) * 100), guarded for requestedPrecise > 0.
Leave the requested model's estimated.max_cost_usd_cents and max_cost_display exactly as today (the Math.ceil conservative value). Only the alternatives' selection, ordering, and savings_pct change.
Investigate the provider-filter question and write one sentence in the final report: is limiting alternatives to active-connection providers intended (suggest only callable models) or should it surface cheaper models the user can't currently call? Recommendation: keep "only callable models" — but confirm, don't change.
