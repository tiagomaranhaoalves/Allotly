# Piece 1 — "Test your key now" button on /redeem

## Why
Every voucher recipient redeems, copies a snippet, pastes into Cursor/Claude.ai, and *hopes*. If it doesn't work, they have no diagnostic and bounce. This button validates the entire chain (auth + tier + provider + budget) in one click, before they wire any AI tool.

## Locked decisions — Piece 1

1. **New endpoint:** `POST /api/v1/test-connection`. Auth: same Bearer key as `/api/v1/chat/completions`. Counts against budget, rate limit, concurrency identically to a real call (because it is one).
2. **Server-side behavior:** the endpoint internally invokes the existing chat completion path with:
   - `model`: cheapest model in the user's allowlist that is currently active across providers
   - `messages`: `[{ role: "user", content: "Reply with the single word 'ok' and nothing else." }]`
   - `max_tokens`: 5
   - `temperature`: 0
3. **Response shape:**
   ```ts
   {
     success: boolean,
     model_used: string,                    // e.g. "claude-haiku-4-5"
     response_text: string,                 // the model's reply, e.g. "ok"
     cost_usd_cents: number,                // small number, typically < 1
     budget: {                              // post-call snapshot
       remaining_usd_cents: number,
       total_usd_cents: number,
       display: { ... }                     // M3a's display block
     },
     latency_ms: number,
     error?: {
       code: 'no_providers_active' | 'no_models_in_tier' | 'budget_exhausted' | 'rate_limited' | 'provider_error' | 'unknown',
       message: string,                     // user-facing
       hint: string,                        // suggested action
     }
   }
   ```
4. **Error mapping** — translate internal errors to user-facing codes:
   - No active provider connections in user's tier → `no_providers_active`, hint: "Contact your admin to configure providers."
   - User's allowlist filters out every active model → `no_models_in_tier`, hint: "Contact your admin to expand your allowed models."
   - Budget already exhausted → `budget_exhausted`, hint: "Request a top-up from your admin via `request_topup` or visit the dashboard."
   - Rate limited → `rate_limited`, hint: "Wait 60 seconds and try again."
   - Upstream provider 5xx → `provider_error`, hint: "Try again in a moment. If it persists, the upstream provider may be down."
   - Anything else → `unknown`, hint: "Contact support if this persists."
5. **No leaking internals.** Don't expose provider names, internal config, error stack traces, or anything not user-actionable.
6. **Frontend: button placement.** New component `client/src/components/redeem/test-key-button.tsx`. Placed inline on `/redeem` post-redemption view, **above** the connector grid. Heading: "✓ Voucher redeemed. Test your key works:". Button label: "Test connection".
7. **Frontend: result display.** Success: green checkmark, "Connection works!", model name, cost ("$0.0001" or "R$ 0,0005" via M3a's formatter), budget remaining (formatted), latency. Failure: red X, error message, hint, retry button.
8. **Idempotent:** repeated clicks each make a real call. No client-side caching of the result. (We're not optimizing — the user might want to re-test after fixing something.)
9. **Optional `_meta.skipBudget`:** **NO**. The endpoint always charges the budget. We're not creating a budget-bypass path; that would be abusable.

## Files — Piece 1

**Create:**

- `client/src/components/redeem/test-key-button.tsx` — button + result display + retry

**Modify:**

2. POST with no active provider connections → 200, `success: false`, `error.code: 'no_providers_active'`

# Piece 3 — New `estimate_cost` MCP tool

## Why
Power-user feature. Lets the host LLM ([Claude.ai](http://Claude.ai), ChatGPT, Gemini, Cursor) preview a `chat` request's max cost before invoking — *"this will cost ~$0.05, proceed?"*. Pairs with the host-LLM-picks-model pattern: now the LLM can also say "Sonnet would cost $X, Haiku $Y, want me to use Haiku?"

## Locked decisions — Piece 3

1. **Tool name:** `estimate_cost`. Registered in `server/lib/mcp/tools/consumption/`.
2. **Description (verbatim, will be pinned via description-hash):**
   > *"Estimate the maximum cost of a chat request before invoking it. Takes the same inputs as `chat` (model, messages, max_tokens) and returns the estimated input tokens, max output tokens, and worst-case cost in the user's currency, plus 2-3 cheaper alternative models if available. Does not invoke the model — purely a preview. Use this when the user asks 'how much will this cost?' or before running expensive long-output requests."*
3. **Input schema:** identical to `chat` tool's input schema except `messages` is required and the tool does not accept tool/vision-specific fields beyond what's needed for token counting (system, max_tokens, model).
4. **Output shape:**
   ```ts
   {
     model: string,
     estimated: {
       input_tokens: number,
       max_output_tokens: number,
       max_cost_usd_cents: number,
       max_cost_display: {                    // M3a-formatted in user's org currency
         currency: string,
         amount: number,
         formatted: string                    // e.g. "$0.05" or "R$ 0,25"
       },
     },
     alternatives: Array<{
       model: string,
       max_cost_usd_cents: number,
       max_cost_display: { currency, amount, formatted },
       savings_pct: number,                   // vs the requested model, e.g. 80 (means 80% cheaper)
     }>,
     disclaimer: "Estimate based on max_tokens; actual cost depends on output length. Token counts are approximate."
   }
   ```
5. **`requiresAuth: true`, `requiredScope: "mcp:read"`, `voucherOnly: false`.** OAuth bearers with `mcp:read` scope can use this. It's a read-only operation that doesn't change state or charge budget.
6. **Annotations:** `title: "Estimate cost of a chat request"`, `readOnlyHint: true`, `destructiveHint: false`, `idempotentHint: true`, `openWorldHint: false`.
7. **Token counting:** reuse the existing token-counting helpers in `server/lib/proxy/cost-utils.ts` (extracted during M4). For input tokens, use the model-specific tokenizer if cheap; otherwise fall back to the proxy's existing approximation. Document the approximation in the response disclaimer.
8. **Alternatives selection:** filter the user's full allowlist for models with the same capability flags as the requested model (vision-capable if input has images, tool-capable if `tools` field present). Sort by cost ascending. Return the top 3 strictly cheaper than the requested model. If fewer than 3 cheaper exist, return what's available. If the requested model is already the cheapest, return an empty alternatives array.
9. **Validation:** if the requested model isn't in the user's allowlist → `McpToolError` with code `ModelNotAllowed` and the existing error-shape pattern.
10. **No charges.** This tool never charges budget, never increments rate-limit counters (it doesn't go upstream). It does count against MCP request limits as one tool call.
11. **Audit:** logs a `mcp_audit_log` row with `tool_name: "estimate_cost"`, `ok: true`, no cost. Standard audit pattern.

**Create:**
- `server/lib/mcp/tools/consumption/estimate-cost.ts` — tool registration + handler
- `tests/mcp-estimate-cost.test.ts` — tool tests

**Modify:**
- `server/lib/mcp/tools/index.ts` — `import "./consumption/estimate-cost"`

# Piece 4 — Budget warnings embedded in `my_status`, `my_budget`, `my_recent_usage`

## Why
No proactive budget alerts today. Users discover they're at 5% only when they ask. Embedded `warning` field in tool responses lets host LLMs surface warnings naturally in conversation without us shipping email/push infrastructure.

## Locked decisions — Piece 4

1. **Helper:** `getBudgetWarning(remainingCents, totalCents, allowlist, currency)` in `server/lib/mcp/budget-warnings.ts`. Returns `Warning | null`.
2. **Thresholds (locked):**
   - `total > 0 && remaining / total < 0.10 && remaining > 0` → `level: 'critical'`
   - `total > 0 && remaining / total < 0.25 && remaining / total >= 0.10` → `level: 'low'`
   - `total > 0 && remaining <= 0` → `level: 'exhausted'`
   - `total === 0` (unlimited) → `null` (no warning)
   - `remaining / total >= 0.25` → `null` (healthy)
3. **`Warning` shape:**
   ```ts
   {
     level: 'low' | 'critical' | 'exhausted',
     message: string,                     // localized using server-side default locale per currency (M3a pattern)
     remaining_pct: number,               // e.g. 8 (means 8%)
     suggestion: {
       text: string,                      // user-facing
       cheapest_model_in_allowlist: string | null,  // e.g. "claude-haiku-4-5"; null if allowlist is empty
       topup_url: string | null,          // direct link to dashboard topup page; null if user can't request topup (e.g. synthetic voucher user without admin)
     },
   }
   ```
4. **Message wording (locked, in English; localize via existing i18n keys):**
   - `low`: "Budget at {remaining_pct}% — running low. {currency_amount_remaining} of {currency_amount_total} left."
   - `critical`: "Budget critically low — only {remaining_pct}% left. {currency_amount_remaining} remaining."
   - `exhausted`: "Voucher fully spent. No budget remaining."
5. **Suggestion text wording (locked):**
   - `low`: "Try cheaper models like {cheapest_model} for routine tasks. Top up here: {topup_url}."
   - `critical`: "Switch to {cheapest_model} immediately or top up: {topup_url}."
   - `exhausted`: "Request a top-up to continue: {topup_url}."
   - When `cheapest_model` is null: drop the model suggestion clause.
   - When `topup_url` is null: replace with "Contact your admin for a top-up."
6. **Where the warning appears:**
   - `my_status` response — `warning?: Warning | null` field at top level
   - `my_budget` response — same `warning?` field at top level
   - `my_recent_usage` response — same `warning?` field at top level
   - `BudgetSnapshot` (the canonical type) gains the `warning?` field — additive, doesn't break callers
7. **Currency-aware:** the formatted amounts in messages use M3a's `formatCurrency` with the org's currency and default locale per currency.
8. **Dashboard surfacing:** if `BudgetSnapshot.warning` is present in any API response the dashboard consumes, render a banner at the top of the relevant page (`overview`, `usage`, `bundles`):
   - `low` → yellow banner
   - `critical` → orange banner
   - `exhausted` → red banner
   - Banner shows `message` + `suggestion.text` + topup CTA button (if `topup_url` is set)
9. **Webhooks unaffected.** No `warning` field in webhook payloads.
10. **Tests deterministic:** when computing `remaining_pct`, round to nearest integer for stability.

**Create:**
