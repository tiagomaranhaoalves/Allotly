# Allotly Proxy — Testing Results & Bugs (Updated)

I ran a full manual test of the proxy endpoint using a voucher key from my terminal. The core proxy infrastructure works — auth, routing, budget headers, and Anthropic translation are solid. But I found several bugs that need fixing. Please check each one, verify if it still exists, and fix if so.

---

## Bug 1 (CRITICAL): Budget is deducted on failed upstream requests

**Evidence:**
I sent a request to `gemini-2.0-flash`. The response was HTTP 502 (Google returned a 404 because the model is deprecated). However, the budget headers showed:
- Before: `x-allotly-budget-remaining: 492`
- After: `x-allotly-budget-remaining: 490` and `x-allotly-requests-remaining` decremented

The proxy reserved budget, forwarded the request, got an error from Google, returned 502 to the client — but **never refunded the reservation**.

**Expected behavior (per spec, Section 9.3 — "ON ERROR/TIMEOUT"):**
```
ON ERROR/TIMEOUT at any point after reservation:
  INCRBY allotly:budget:{membershipId} estimatedTotalCents  ← full refund
  DECR allotly:concurrent:{membershipId}  ← release slot
  DEL allotly:req:{requestId}
  Return appropriate error to client
```

**What to fix:**
Check the proxy error handling path. After forwarding to the provider, if the response is a 4xx or 5xx, the reserved budget must be refunded via `INCRBY` before returning the error to the client. The request counter should also not be decremented for failed requests.

Note: On a second failed Google request, the budget did NOT decrease (stayed at 490), so the bug may be inconsistent — possibly only triggered on certain error types or race conditions. Please review all error paths in the proxy handler.

## Bug 3 (HIGH): Google request translation appears broken

**Evidence:**
The error message from Google says: `"models/gemini-2.5-flash-preview-05-20 is not found for API version v1beta, or is not supported for generateContent"`

Two issues here:

1. **Wrong API version:** The proxy is using `v1beta`. Check if it should be using `v1` instead, or if the model IDs need to match the version.

2. **Model ID format mismatch:** Google's generateContent API expects the model in the URL path like:
   ```
   POST https://generativelanguage.googleapis.com/v1/models/gemini-2.0-flash:generateContent
   ```
   The proxy may be passing the full Allotly model ID (with preview suffixes) directly into the URL, when Google expects a different format.

**What to fix:**
1. Check the Google provider adapter (`/lib/providers/google.ts`) — verify the target URL construction and API version
2. Verify what model IDs Google actually accepts by calling their ListModels endpoint
3. Ensure the model ID in the seed data matches exactly what Google expects in the generateContent URL

## Bug 4 (MEDIUM): Upstream error messages are truncated

**Evidence:**
OpenAI 429 error response:
```json
{
  "message": "OPENAI returned 429: {\"error\":{\"message\":\"You exceeded your current quota, please check your plan and billing details. For more information on this error, read the docs: https://platform.openai.com/docs/gu"
}
```

The message cuts off mid-URL.

**What to fix:**
1. Check if there's a character limit when capturing upstream error bodies
2. Either: capture the full upstream error message, or parse the upstream JSON and extract just the `message` field for a clean summary like: `"OPENAI returned 429: You exceeded your current quota, please check your plan and billing details."`

## Bug 5 (MEDIUM — needs confirmation): Pricing values may be 10x too high

**Evidence:**
The `/api/v1/models` response shows `gpt-4o-mini` with:
- `input_price_per_m_tok: 150`
- `output_price_per_m_tok: 600`

OpenAI's actual pricing for gpt-4o-mini is $0.15 input / $0.60 output per 1M tokens. If the field unit is "cents per 1M tokens", the values should be `15` and `60`, not `150` and `600`.

**Impact:** If real, every request costs users 10x more budget than it should. At small token counts the `ceil()` minimum of 1 cent masks this, but larger requests would be significantly overcharged.

**What to fix:**
1. Clarify the unit convention for `input_price_per_m_tok` — is it cents, tenths of cents, or hundredths of cents?
2. Compare every seeded price against current provider pricing pages
3. If 10x off, update all seed values and reconsider whether existing budget deductions need correction
