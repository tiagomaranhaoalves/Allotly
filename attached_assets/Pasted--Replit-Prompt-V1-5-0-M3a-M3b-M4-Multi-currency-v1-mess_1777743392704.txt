# Sub-project M3a — Multi-currency budget display

## Why
Allotly's user base is global (Brazil, EU, UK significant). Users currently see "$49.94 of $50.00" regardless of locale. M3a makes budgets display in the user's organisation's chosen currency with locale-correct formatting, while internal accounting stays USD-cents. No proxy hot-path changes.

## Locked decisions — M3a

1. **Supported currencies:** `USD`, `GBP`, `EUR`, `BRL` (4 total). Add as a Postgres enum `CURRENCY`. Default `USD`.
2. **Currency lives on `organizations`.** New column `currency CURRENCY NOT NULL DEFAULT 'USD'`. All members of an org share currency. Voucher recipients (synthetic users) inherit the issuing org's currency.
3. **FX storage:** new table `fx_rates` (one row per non-USD currency, upserted daily by cron). Schema below.
4. **FX source:** [exchangerate.host](https://exchangerate.host) free API (no key). Endpoint: `https://api.exchangerate.host/latest?base=USD&symbols=GBP,EUR,BRL`.
5. **Cron schedule:** daily at 03:00 UTC. Job name `fx-rates-refresh`. Add to existing scheduler in `server/lib/jobs/scheduler.ts`.
6. **Hardcoded fallback rates** in `server/lib/fx/fallback-rates.ts`, used only when the FX cron has never succeeded AND the live API call fails:
   ```ts
   export const FALLBACK_FX_RATES: Record<NonUsdCurrency, number> = {
     GBP: 0.79,
     EUR: 0.92,
     BRL: 5.20,
   };
   ```
7. **Conversion helper:** `convertUsdCentsToCurrency(cents, currency)` returns `{ amount: number, currency: Currency, rate: number, source: 'live' | 'fallback', fetchedAt: ISO8601 | null }`. Always returns `rate: 1, source: 'live'` for USD.
8. **`BudgetSnapshot` shape change.** The existing tool responses for `my_budget`, `my_status`, `my_recent_usage` add a `display` block:
   ```ts
   display: {
     currency: 'USD' | 'GBP' | 'EUR' | 'BRL',
     remaining: number,        // converted, e.g. 488.90
     total: number,            // converted
     used: number,             // converted
     rate: number,             // 1 for USD; e.g. 0.79 for GBP
     formatted: {              // server-formatted using default locale-per-currency
       remaining: string,      // "R$ 254,23" or "$49.94" or "£39.46" etc.
       total: string,
       used: string,
     },
   }
   ```
   The existing `remainingCents`, `totalCents`, etc. fields stay — clients keep working. `display` is purely additive.
9. **Default locale per currency** for server-side `Intl.NumberFormat`:
   - `USD` → `en-US`
   - `GBP` → `en-GB`
   - `EUR` → `de-DE`
   - `BRL` → `pt-BR`
10. **Frontend formatting:** dashboard pages that show budget should prefer `Intl.NumberFormat(navigator.language || defaultLocale, { style: 'currency', currency })`. Use the server's `formatted` strings as fallback if `Intl.NumberFormat` errors (some locales/browsers).
11. **Voucher emails:** the existing budget-mention text uses `display.formatted.total` so recipients see their currency. Email rendering uses server-side default locale (no browser context).
12. **Webhooks stay USD.** External integrations expect stable USD format.
13. **Currency settings UI.** New section "Display currency" on `/dashboard/settings` with a dropdown (USD/GBP/EUR/BRL). Org admins only. Saves via `PATCH /api/organizations/:orgId/settings { currency }`. Non-admins see read-only label.
14. **Migration safety.** New column has a default; existing rows backfill to `'USD'` automatically. No data migration needed.

## Schema — M3a

In `shared/schema.ts`:

```ts
// New enum
export const currencyEnum = pgEnum("currency", ["USD", "GBP", "EUR", "BRL"]);

// Add to organizations table:
currency: currencyEnum("currency").notNull().default("USD"),

// New table
export const fxRates = pgTable("fx_rates", {
  currency: currencyEnum("currency").primaryKey(),  // not USD; only stores non-USD
  rateFromUsd: decimal("rate_from_usd", { precision: 12, scale: 6 }).notNull(),
  source: text("source", { enum: ["live", "fallback"] }).notNull(),
  fetchedAt: timestamp("fetched_at").defaultNow().notNull(),
});
```

# Sub-project M3b — `/api/v1/messages` Anthropic-native endpoint

## Why
Today, developers using the Anthropic SDK can't point `baseURL` at Allotly because we only expose `/api/v1/chat/completions` (OpenAI-compat). M3b adds the missing Anthropic-native endpoint so they can drop Allotly in front of any Anthropic SDK app and get budget control without code changes.

## Locked decisions — M3b

1. **Endpoint:** `POST /api/v1/messages`. Anthropic-native request/response shape.
2. **Reuses every V1 safeguard:** auth (Bearer key/voucher/OAuth), budget, concurrency, rate limit, audit, error mapping. Same middleware chain as `/api/v1/chat/completions`.
3. **No code in `processChatCompletion`.** Add a sibling function `processMessages` in `server/lib/proxy/handler-messages.ts` (new file). Share helpers with handler.ts (auth, budget, error mapping) but not the request/response loop.
4. **Provider routing:**
   - **Anthropic providers** (Claude direct API, AWS Bedrock if applicable): pass through verbatim — request body straight to upstream, response body straight back to client.
   - **OpenAI / Azure OpenAI providers:** translate Anthropic format to OpenAI Chat Completions on the way upstream; translate OpenAI response back to Anthropic Messages format on the way back.
   - **Google providers:** translate Anthropic to Google Gemini format and back.
   - All translations live in `server/lib/proxy/translate.ts` — extend the existing translation functions; do not create a new module.
5. **Streaming support:** if request body has `stream: true`, return SSE per Anthropic's standard event types: `message_start`, `content_block_start`, `content_block_delta`, `content_block_stop`, `message_delta`, `message_stop`, `ping`, `error`. Reuse `server/lib/proxy/streaming.ts` plumbing; adapt the framing for Anthropic event types.
6. **Default model selection:** if request omits `model`, pick the cheapest allowed Anthropic-format-compatible model (same logic as `selectDefaultModel` in `chat.ts`, but biased to Anthropic providers when available — using Anthropic-native is meaningless if we then route to GPT).
7. **Tool use / vision / system prompts:** all passed through as-is. Any feature Anthropic supports natively must work via this endpoint.
8. **Prompt caching, thinking, citations:** if request includes Anthropic-specific fields (`cache_control`, `thinking`, `citations`), pass them through to Anthropic-native upstreams. For non-Anthropic upstreams, drop those fields with a warning header `X-Allotly-Dropped-Fields: cache_control,thinking`.
9. **Response headers:** include the existing `X-Allotly-Budget-Remaining-USD-Cents`, `X-Allotly-Budget-Total-USD-Cents`, `X-Allotly-Effective-Model`, `X-Allotly-Request-ID`. Add `X-Allotly-Native-Format: anthropic` for parity with the implicit `openai` on `/chat/completions`.
10. **Error mapping:** Anthropic errors stay Anthropic-shaped (`{ type: 'error', error: { type, message } }`). OpenAI/Google errors are translated to Anthropic error shape on the way back.
11. **No webhook changes.** This is a proxy endpoint, not a billing event.

- `server/lib/proxy/messages-schemas.ts` — Zod schemas for Anthropic Messages request/response
- `tests/proxy-messages.test.ts` — endpoint tests

# Sub-project M4 — MCP `chat` tool streaming

## Why
The MCP `chat` tool currently forces `stream: false` and buffers full responses. Long generations (>30s) often time out at the MCP client because no progress signal goes back. Streaming makes `chat` feel native and unblocks generation-heavy use cases. **Strictly additive — `processChatCompletion` is not touched.**

## Locked decisions — M4

1. **Strictly additive design.** New function `processChatCompletionStreaming` in a new file `server/lib/proxy/handler-streaming.ts`. The existing `processChatCompletion` is **not edited**. The chat tool branches on whether `_meta.progressToken` is present.
2. **Feature flag:** environment variable `MCP_STREAMING_ENABLED`. Default `false` for the first week post-deploy. When `false`, all chat requests use today's buffered path regardless of `_meta.progressToken`. When `true`, chat requests with `_meta.progressToken` use the streaming path. Must be tested in both modes.
3. **Affects only the `chat` MCP tool.** No other MCP tools change. `compare_models` does not stream (it's already a multi-call aggregation).
4. **MCP transport extension.** [server/lib/mcp/transport.ts](server/lib/mcp/transport.ts) gains the ability to send mid-call `notifications/progress` JSON-RPC messages, then a final `tools/call` response, all over a single HTTP response. Use Streamable HTTP framing (newline-delimited JSON; one JSON message per line; `Content-Type: application/x-ndjson`).
5. **Activation path:** when `tools/call` body contains `params._meta.progressToken: <string>` AND the tool name is `chat` AND `MCP_STREAMING_ENABLED=true`, the handler enters the streaming branch. Every other path (tool != chat, no progressToken, flag off) keeps today's behavior bit-for-bit.
6. **Progress notification payload shape:**
   ```ts
   {
     jsonrpc: "2.0",
     method: "notifications/progress",
     params: {
       progressToken: "<echo from request>",
       progress: <number>,        // monotonically increasing — token count emitted so far
       total: <number | undefined>, // from request max_tokens if specified
       message: <chunk-text-since-last-progress>,
     }
   }
   ```
7. **Final response.** After upstream stream ends, send a single `tools/call` response message with the same shape today's chat tool returns: `{ content, model, finish_reason, usage, ... }`. Same `id` as the original `tools/call` request.
8. **Budget reservation.** Before initiating upstream stream, compute pessimistic max cost: `inputTokens * inputPricePerMTok / 1000 + maxTokens * outputPricePerMTok / 1000` (cents). Reserve via existing budget ledger. On stream end, settle to actual usage; refund the unused reservation atomically. If `max_tokens` is omitted, use the model's max output tokens from the pricing table.
9. **Concurrency:** streaming chat counts against the same per-membership concurrency limit as non-streaming chat. Acquire a slot before reservation; release in the cleanup `finally`.
10. **Rate limit:** same per-membership rate limit. Counted as one request regardless of stream length.
11. **Audit log.** Same `mcp_audit_log` row, populated after the stream ends with actual token counts. Add a new column `streamed BOOLEAN NOT NULL DEFAULT false`. Migration required.
12. **Disconnect handling:**
    - Client disconnects mid-stream → close upstream connection, settle budget to actual-tokens-consumed-so-far, release concurrency slot, write audit row with `ok: false, errorCode: -32099, finishReason: 'client_disconnected'`.
    - Upstream provider drops mid-stream → emit one final `notifications/progress` with `error` field, then `tools/call` error response, settle budget, release slot, audit.
    - Server crashes mid-stream → reservation TTL (5 minutes) eventually expires, ledger reconciles. Acceptable failure mode.
13. **No edits to `processChatCompletion`.** Period. If any helper function needs to be shared (e.g. token cost calculation), extract it to `server/lib/proxy/cost-utils.ts` as a new file and import from both functions. Don't move logic out of the existing function.
14. **Compatibility with M3b.** The streaming work in M3b (`/api/v1/messages` with `stream: true`) is HTTP SSE; M4 streaming is MCP `notifications/progress`. Different framings, different consumers. Code-share opportunities are limited to upstream HTTP request handling — extract those into `server/lib/proxy/upstream-stream.ts` (new) and use from both M3b's streaming path and M4's streaming path. The transport-level framing (SSE vs ndjson) lives in the respective endpoint handlers.
