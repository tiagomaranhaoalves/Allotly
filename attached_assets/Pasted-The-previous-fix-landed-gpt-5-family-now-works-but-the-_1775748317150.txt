The previous fix landed (gpt-5 family now works), but the o-series fix didn't take effect in production. Cause: the database stores azureApiVersion="2024-10-21" on the user's existing Azure connection, which short-circuits the new default via `connection.azureApiVersion || DEFAULT`. The bumped default never kicks in for existing connections.

This patch fixes that AND improves the UX so users don't have to know what api-version means.

PART A — Per-model minimum api-version (the actual fix)
================================================================

In server/lib/providers/azure-openai.ts (or wherever DEFAULT_AZURE_API_VERSION lives — create the file if needed), add:

  export const DEFAULT_AZURE_API_VERSION = "2024-12-01-preview";

  // Models that require api-version >= 2024-12-01-preview
  const REASONING_MODEL_PATTERN = /^(o1|o3|o4|gpt-5)/i;
  const REASONING_MIN_API_VERSION = "2024-12-01-preview";

  // Returns the effective api-version for a given model, taking the
  // newer of (user-stored value, model-required minimum, default).
  export function effectiveAzureApiVersion(modelId: string, userValue?: string | null): string {
    const candidates = [userValue, DEFAULT_AZURE_API_VERSION].filter(Boolean) as string[];
    if (REASONING_MODEL_PATTERN.test(modelId)) {
      candidates.push(REASONING_MIN_API_VERSION);
    }
    // Date strings sort lexically; pick the newest
    return candidates.sort().reverse()[0];
  }

Then update server/lib/proxy/handler.ts:380 to use this helper when constructing azureContext:

  apiVersion: effectiveAzureApiVersion(modelId, connection.azureApiVersion),

(modelId here is the model from the request after the "azure/" prefix is stripped — the same value that flows into translate.ts's azureContext.modelId.)

Same change in:
- server/routes.ts:734 (azure connection test on save) — for the test, use a placeholder model like "gpt-4o" since we don't know which model will be called
- server/lib/jobs/provider-validation.ts:24 — same, use a placeholder

Leave server/lib/proxy/translate.ts:175 alone — it just consumes azureContext.apiVersion which is now always correct.

PART C — One-time migration for existing connections
================================================================

Add a migration (or a one-time on-startup cleanup, your call — migration is cleaner) that does:

  UPDATE provider_connections
     SET azure_api_version = NULL
   WHERE provider = 'azure_openai'
     AND azure_api_version = '2024-10-21';

This unblocks the user's existing connection without requiring them to re-edit it. The reasoning: 2024-10-21 was the old auto-default — no user explicitly chose it as a compliance pin. Anyone who genuinely needs that exact version can re-set it after the migration runs.

If you're uncomfortable with a destructive migration, alternative: just rely on the per-model override from PART A (which already works — the migration is convenience, not correctness).
