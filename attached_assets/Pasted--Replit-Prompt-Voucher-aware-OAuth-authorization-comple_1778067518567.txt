# Replit Prompt — Voucher-aware OAuth authorization (complete)

You are extending `/oauth/authorize` so voucher recipients and API-key holders can complete the OAuth handshake without an email+password account. Today an unauthenticated user landing on `/oauth/authorize` is redirected to `/login` (dead end for voucher recipients), and a synthetic voucher user with an active session is redirected to `/oauth/claim-account` (forces a "claim" detour). Both gates come down. The MCP transport's three-bearer-kinds auth (api key / voucher / OAuth) is unchanged.

This is **production-grade work** — security-critical, no schema changes, no refactor of unrelated files. **All locked decisions below are non-negotiable. One end-of-task report.**

## Cross-cutting locked decisions (read first)

1. **No schema changes.** Do not add `is_anonymous`. Do not make email nullable. Do not create a partial unique index. The codebase already has `users.isVoucherUser: boolean` (`shared/schema.ts:84`) and synthetic voucher users are already minted with `voucher-{code}-{rand}@allotly.local` emails by the existing `/api/vouchers/redeem` flow.
2. **No edits to `processChatCompletion`** or any proxy handler.
3. **No edits to OAuth token, consent, register, revoke handlers.** Only `/oauth/authorize` GET and a new POST `/oauth/authorize/credential`.
4. **Voucher recipients are a spectrum**, not a single type:
   - Synthetic voucher user (`isVoucherUser=true`, `@allotly.local` email, random unrecoverable password)
   - Real-account voucher user (regular email + password, also has voucher membership via `claim-from-voucher` or having redeemed with their own email)
   - Tests must cover both kinds. All three credential paths (password / voucher / api_key) must resolve to the **same `User` row** when they belong to the same person.
5. **Currency / display helpers** are at `server/lib/currency.ts` (`convertFromUsdCents`, `formatMoney`, `buildDisplayBlock`, `getOrgCurrency`) — no surface in this task uses them; mention only if you find a place that needs them.
6. **All existing 700+ tests must pass unchanged.**
7. **Pre-existing TS errors** in `server/routes.ts`, `server/webhookHandlers.ts`, `server/lib/plan-limits.ts`, `server/lib/jobs/*` are out of scope.
8. **CSRF token** is already on the session as `_oauthCsrf` — see `authorize.ts::newCsrfToken`. Reuse it for the credential form. Mint a fresh one if absent.
9. **Existing rate limiters** (`redeemLimiter`, `loginLimiter` in `server/lib/rate-limiter.ts`) are reused, not replaced.

## Architecture

```
GET /oauth/authorize  (no session OR stale session)
  └─ render credential form HTML  →  POST /oauth/authorize/credential
                                       ├─ password path  → comparePasswords  →  set session
                                       ├─ voucher path   → redeemVoucherInline (instant: true)  →  set session
                                       └─ api_key path   → lookupApiKey  →  set session
                                       └─ on success → 302 → /oauth/authorize?<original query>
                                       └─ on failure → 200 + same form + generic error
GET /oauth/authorize  (session is now set — including synthetic voucher users)
  └─ existing OAuth flow continues normally — no claim-account redirect
```

### 1. `server/lib/vouchers/redeem-inline.ts`

Extract the entire body of `POST /api/vouchers/redeem` (currently `server/routes.ts:3840-4000`) into a pure helper:

```ts
export interface RedeemInlineInput {
  code: string;        // ALLOT-XXXX-XXXX-XXXX, case-insensitive (normalised internally)
  email?: string;
  name?: string;
  password?: string;
  instant?: boolean;   // when true OR email missing → synthetic user with @allotly.local email
}

export type RedeemInlineErrorCode =
  | 'invalid_voucher' | 'voucher_expired' | 'fully_redeemed'
  | 'team_not_found' | 'plan_limit' | 'bundle_exhausted' | 'internal_error';

export type RedeemInlineOutcome =
  | { ok: true; user: User; membership: TeamMembership; voucher: Voucher; team: Team;
      apiKey: string; keyPrefix: string; isSynthetic: boolean }
  | { ok: false; code: RedeemInlineErrorCode; message: string };

export async function redeemVoucherInline(input: RedeemInlineInput): Promise<RedeemInlineOutcome>;
```

The helper must replicate the existing endpoint's side-effects exactly:
- Validate voucher (ACTIVE, not expired, not fully redeemed)
- Look up `team`, run `checkPlanLimit(orgId, "member", teamId)`
- Mint synthetic email `voucher-{code8}-{rand6}@allotly.local` and 32-hex random password when `instant || !email`. Use `crypto.randomBytes(...)` (NOT `Math.random()`) for both — same behaviour, slightly stronger entropy.
- `storage.createUser({...isVoucherUser: true})`
- `storage.createMembership({accessType: 'VOUCHER', voucherRedemptionId: voucher.id, periodStart: now, periodEnd: voucher.expiresAt, status: 'ACTIVE'})`
- `storage.createVoucherRedemption(...)` and `storage.updateVoucher(...)` to bump `currentRedemptions`; flip status to `FULLY_REDEEMED` if capacity reached
- For bundle-backed vouchers, atomic CAS `db.update(voucherBundles)...returning()` checking `usedRedemptions < totalRedemptions` AND `status = 'ACTIVE'`. Empty result → return `bundle_exhausted` outcome. Otherwise prime `REDIS_KEYS.bundleRequests(bundle.id)` if absent and `redisIncr(REDIS_KEYS.bundleRedemptions(bundle.id))`.
- `redisSet(REDIS_KEYS.budget(membership.id), String(voucher.budgetCents))`
- `generateAllotlyKey()` → `storage.createAllotlyApiKey(...)`
- `storage.createAuditLog({action: 'voucher.redeemed', metadata: {code, email, isSynthetic}})`
- Best-effort send `emailTemplates.voucherRedeemed(...)` to `team.adminId`'s user email (wrap in try/catch — do NOT fail the redemption on email error).

The helper does NOT touch `req.session`.

### 3. `server/lib/auth/api-key-lookup.ts`

Pure validate-and-resolve helper for `allotly_sk_…` keys. No HTTP coupling, no Redis caching (this runs at most a handful of times per OAuth login; the proxy hot path keeps its existing cache).

```ts
export type ApiKeyLookupErrorCode =
  | 'invalid_format' | 'not_found' | 'revoked' | 'membership_missing'
  | 'budget_exhausted' | 'suspended' | 'expired' | 'period_expired';

export type ApiKeyLookupOutcome =
  | { ok: true; user: User; membership: TeamMembership; apiKey: AllotlyApiKey }
  | { ok: false; code: ApiKeyLookupErrorCode };

export async function lookupApiKey(plaintext: string): Promise<ApiKeyLookupOutcome>;
```

Validation order, mirroring `server/lib/proxy/safeguards.ts::authenticateKey`:
1. Format regex `/^allotly_sk_[a-zA-Z0-9_-]{16,}$/` after trim
2. SHA-256 hex of the trimmed plaintext → `storage.getApiKeyByHash`
3. `apiKey.status === 'ACTIVE'` else `revoked`
4. `storage.getMembership(apiKey.membershipId)` else `membership_missing`
5. `membership.status` checks: `BUDGET_EXHAUSTED | SUSPENDED | EXPIRED`
6. `new Date(membership.periodEnd) < new Date()` → `period_expired`
7. `storage.getUser(apiKey.userId)` else `not_found`

### 5. `server/lib/oauth/authorize-credential.ts`

POST handler. **One generic error message** for ALL failure modes (no enumeration aid in user-visible body — audit log captures the precise reason):
> *"Invalid credential. Check your code, key, or password and try again."*

Exception: CSRF mismatch → "Your session expired. Please try again." Open-redirect attempt → "This sign-in link is missing context. Start over from the application that sent you here."

Validation order:
1. Validate `oauth_continue`: must start with `/oauth/authorize`, must NOT contain `://`, must NOT start with `//`. On failure render form with "missing context" error.
2. Mint `req.session._oauthCsrf` if absent (using `crypto.randomBytes(16).toString('hex')`).
3. Validate `csrf` matches `req.session._oauthCsrf` via `crypto.timingSafeEqual`. On mismatch render "session expired" form.
4. Switch on `credential_type`:
   - **`password`**: `storage.getUserByEmail(email.toLowerCase())` → `comparePasswords(password, user.passwordHash)` → set `session.userId/orgId/orgRole`; best-effort `storage.updateUser(user.id, {lastLoginAt: new Date()})`.
   - **`voucher`**: regex `^ALLOT-[A-Z0-9]{4}-[A-Z0-9]{4}-[A-Z0-9]{4}$` after trim+upper → `redeemVoucherInline({code, instant: true})` → set `session.userId = outcome.user.id`, `session.orgId = outcome.voucher.orgId`, `session.orgRole = "MEMBER"`.
   - **`api_key`**: regex `^allotly_sk_[a-zA-Z0-9_-]{16,}$` after trim → `lookupApiKey(...)` → set `session.userId/orgId/orgRole` from the resolved user.
5. On success: `storage.createAuditLog({action: 'oauth.credential.granted', metadata: {credentialType, voucherId?, apiKeyId?, synthetic?}})` then `res.redirect(302, oauth_continue)`.
6. On any failure (after CSRF passes): re-render the form with the generic error and the previously-active tab preserved. Audit log row with `action: 'oauth.credential.rejected'` (best-effort; do not block).

Render uses `Content-Security-Policy: default-src 'self'; style-src 'self' 'unsafe-inline'; script-src 'none'; img-src 'self'; form-action 'self'`. Status 200 (UI page that re-asks for credentials, not a protocol error). `Cache-Control: no-store`.

### `server/lib/oauth/index.ts`

Mount the new POST route with rate limiting. The middleware picks the right limiter based on `credential_type`:

```ts
import type { Request, Response, NextFunction } from "express";
import { redeemLimiter, loginLimiter } from "../rate-limiter";
import { authorizeCredentialHandler } from "./authorize-credential";

function credentialRateLimiter(req: Request, res: Response, next: NextFunction): void {
  const t = req.body?.credential_type;
  if (t === "voucher" || t === "api_key") return redeemLimiter(req, res, next);
  return loginLimiter(req, res, next);
}

// inside mountOAuth(app):
app.post("/oauth/authorize/credential", credentialRateLimiter, authorizeCredentialHandler);
```
