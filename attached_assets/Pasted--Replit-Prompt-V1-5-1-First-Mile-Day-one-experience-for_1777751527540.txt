# Piece 1 — "Test your key" button on /redeem AND /dashboard/connect

## Why
Every Allotly key user — voucher recipient, TEAM admin, TEAM member — at some point copies a snippet, pastes it into Cursor/Claude.ai, and *hopes* it works. If it doesn't, there's no diagnostic, and they bounce. This button validates the entire chain (auth + tier + provider + budget) in one click.

Voucher recipients see this on `/redeem` immediately after redemption. TEAM users see it on `/dashboard/connect` whenever they need to verify their setup.

## Locked decisions — Piece 1

1. **New endpoint:** `POST /api/v1/test-connection`. Auth: same Bearer key as `/api/v1/chat/completions`. Counts against budget, rate limit, concurrency identically to a real call (because it is one). Universal — works for any bearer kind (API key, voucher, OAuth bearer).
2. **Server-side behavior:** the endpoint internally invokes the existing chat completion path with:
   - `model`: cheapest model in the user's allowlist that is currently active across providers
   - `messages`: `[{ role: "user", content: "Reply with the single word 'ok' and nothing else." }]`
   - `max_tokens`: 5
   - `temperature`: 0
3. **Response shape:**
   ```ts
   {
     success: boolean,
     model_used: string,                    // e.g. "claude-haiku-4-5"
     response_text: string,                 // the model's reply, e.g. "ok"
     cost_usd_cents: number,                // small number, typically < 1
     budget: {                              // post-call snapshot
       remaining_usd_cents: number,
       total_usd_cents: number,
       display: { ... }                     // M3a's display block — currency-aware
     },
     latency_ms: number,
     error?: {
       code: 'no_providers_active' | 'no_models_in_tier' | 'budget_exhausted' | 'rate_limited' | 'provider_error' | 'unknown',
       message: string,                     // user-facing
       hint: string,                        // suggested action; varies by user type per below
     }
   }
   ```
4. **Error mapping** — translate internal errors to user-facing codes. **Hint text branches by user type** to give actionable guidance:
   - `no_providers_active`:
     - TEAM admin: "Configure providers in `/dashboard/providers`."
     - TEAM member: "Contact your team admin to configure providers."
     - Voucher recipient: "Contact the admin who issued your voucher."
   - `no_models_in_tier`:
     - TEAM admin: "Adjust your tier's model allowlist in `/dashboard/teams`."
     - TEAM member: "Contact your team admin to expand your allowed models."
     - Voucher recipient: "Contact the issuing admin to expand allowed models."
   - `budget_exhausted`:
     - TEAM admin: "Top up via `/dashboard/billing`."
     - TEAM member: "Contact your team admin for a budget increase."
     - Voucher recipient: "Run `request_topup` from your AI client, or contact the issuing admin."
   - `rate_limited`: (universal) "Wait 60 seconds and try again."
   - `provider_error`: (universal) "Try again in a moment. If this persists, the upstream provider may be down."
   - `unknown`: (universal) "Contact support if this persists."
5. **No leaking internals.** Don't expose provider names, internal config, or stack traces.
6. **Frontend: dual placement.**
   - `client/src/components/redeem/test-key-button.tsx` — new component, used in both surfaces
   - **On `/redeem`:** placed **above** the connector grid in the post-redemption view. Heading: "✓ Voucher redeemed. Test your key works:". Button label: "Test connection".
   - **On `/dashboard/connect`:** placed **above** the existing connector grid, near the page heading. Heading: "Test your key works:". Button label: "Test connection". Visible to any logged-in user (TEAM admin or member) regardless of role.
7. **Frontend: result display.** Success: green checkmark, "Connection works!", model name, cost (formatted in user's org currency via M3a), budget remaining (formatted), latency. Failure: red X, error message, hint, retry button.
8. **Idempotent.** Repeated clicks each make a real call. No client-side caching of the result.
9. **No budget-bypass mode.** The endpoint always charges the budget. We're not creating an abuse vector.

# Piece 3 — New `estimate_cost` MCP tool

## Why
Power-user feature. Lets the host LLM (Claude.ai, ChatGPT, Gemini, Cursor) preview a `chat` request's max cost before invoking — *"this will cost ~$0.05, proceed?"*. Universal — works for any bearer kind, any user type. Pairs with the host-LLM-picks-model pattern: now the LLM can also say "Sonnet would cost $X, Haiku $Y, want me to use Haiku?"

## Locked decisions — Piece 3

1. **Tool name:** `estimate_cost`. Registered in `server/lib/mcp/tools/consumption/`.
2. **Description (verbatim, will be pinned via description-hash):**
   > *"Estimate the maximum cost of a chat request before invoking it. Takes the same inputs as `chat` (model, messages, max_tokens) and returns the estimated input tokens, max output tokens, and worst-case cost in the user's currency, plus 2-3 cheaper alternative models if available. Does not invoke the model — purely a preview. Use this when the user asks 'how much will this cost?' or before running expensive long-output requests."*
3. **Input schema:** identical to `chat` tool's input schema except `messages` is required and the tool does not accept tool/vision-specific fields beyond what's needed for token counting (system, max_tokens, model).
4. **Output shape:**
   ```ts
   {
     model: string,
     estimated: {
       input_tokens: number,
       max_output_tokens: number,
       max_cost_usd_cents: number,
       max_cost_display: {                    // M3a-formatted in user's org currency
         currency: string,
         amount: number,
         formatted: string                    // e.g. "$0.05" or "R$ 0,25"
       },
     },
     alternatives: Array<{
       model: string,
       max_cost_usd_cents: number,
       max_cost_display: { currency, amount, formatted },
       savings_pct: number,                   // vs the requested model, e.g. 80 (means 80% cheaper)
     }>,
     disclaimer: "Estimate based on max_tokens; actual cost depends on output length. Token counts are approximate."
   }
   ```
5. **`requiresAuth: true`, `requiredScope: "mcp:read"`, `voucherOnly: false`.** OAuth bearers with `mcp:read` scope can use this. Read-only — doesn't change state, doesn't charge budget.
6. **Annotations:** `title: "Estimate cost of a chat request"`, `readOnlyHint: true`, `destructiveHint: false`, `idempotentHint: true`, `openWorldHint: false`.
7. **Token counting:** reuse the existing token-counting helpers in `server/lib/proxy/cost-utils.ts` (extracted during M4). For input tokens, use the model-specific tokenizer if cheap; otherwise use the proxy's existing approximation. Document the approximation in the response disclaimer.
8. **Alternatives selection:** filter the user's full allowlist for models with the same capability flags as the requested model (vision-capable if input has images, tool-capable if `tools` field present). Sort by cost ascending. Return the top 3 strictly cheaper than the requested model. If fewer than 3 cheaper exist, return what's available. If the requested model is already the cheapest, return an empty alternatives array.
9. **Validation:** if the requested model isn't in the user's allowlist → `McpToolError` with code `ModelNotAllowed`.
10. **No charges, no rate-limit charge.** This tool never charges budget, never increments rate-limit counters (it doesn't go upstream). Counts as one MCP call against MCP-level limits.
11. **Audit:** logs `mcp_audit_log` row with `tool_name: "estimate_cost"`, `ok: true`, no cost.

**Modify:**

# Piece 4 — Budget warnings embedded in `my_status`, `my_budget`, `my_recent_usage`

## Why
No proactive budget alerts today. Users discover they're at 5% only when they ask. Embedded `warning` field in tool responses lets host LLMs surface warnings naturally in conversation. Applies to every user type, with **suggestion text branched by user type** so each user gets actionable guidance.

## Locked decisions — Piece 4

1. **Helper:** `getBudgetWarning(remainingCents, totalCents, membership, allowlist, currency)` in `server/lib/mcp/budget-warnings.ts`. Returns `Warning | null`.
2. **Thresholds (locked):**
   - `total > 0 && remaining / total < 0.10 && remaining > 0` → `level: 'critical'`
   - `total > 0 && remaining / total < 0.25 && remaining / total >= 0.10` → `level: 'low'`
   - `total > 0 && remaining <= 0` → `level: 'exhausted'`
   - `total === 0` (unlimited) → `null`
   - `remaining / total >= 0.25` → `null` (healthy)
3. **`Warning` shape:**
   ```ts
   {
     level: 'low' | 'critical' | 'exhausted',
     message: string,                     // localized using server-side default locale per currency (M3a pattern)
     remaining_pct: number,               // e.g. 8 (means 8%)
     suggestion: {
       text: string,                      // user-facing, branched by user type
       cheapest_model_in_allowlist: string | null,  // e.g. "claude-haiku-4-5"; null if allowlist is empty
       topup_url: string | null,          // direct dashboard link only when user can act on it; null otherwise
       topup_via_mcp_tool: string | null, // e.g. "request_topup"; non-null only for voucher recipients
     },
   }
   ```
4. **Suggestion branching by user type** (the membership row determines this):
   - **TEAM admin** (`accessType === 'TEAM'` AND role in [`OWNER`, `TEAM_ADMIN`]):
     - `topup_url`: `/dashboard/billing`
     - `topup_via_mcp_tool`: `null`
     - Suggestion text includes "Top up here: /dashboard/billing." Admin can act directly.
   - **TEAM member** (`accessType === 'TEAM'`, other role):
     - `topup_url`: `null`
     - `topup_via_mcp_tool`: `null`
     - Suggestion text uses "Contact your team admin to increase the budget." Don't dangle a topup link they can't action.
   - **Voucher recipient** (`accessType === 'VOUCHER'`):
     - `topup_url`: `null`
     - `topup_via_mcp_tool`: `"request_topup"`
     - Suggestion text uses "Run `request_topup` to ask the issuing admin for more budget." Routes through the existing voucher flow.
5. **Message wording (locked, English; localize via i18n keys for `low`, `critical`, `exhausted`):**
   - `low`: "Budget at {remaining_pct}% — running low. {currency_amount_remaining} of {currency_amount_total} left."
   - `critical`: "Budget critically low — only {remaining_pct}% left. {currency_amount_remaining} remaining."
   - `exhausted`: "Voucher fully spent." (or "Budget fully spent." for TEAM users)
6. **Suggestion text wording per user type** (locked):
   - **TEAM admin, low/critical:** "Try cheaper models like {cheapest_model} for routine tasks. Top up here: {topup_url}."
   - **TEAM admin, exhausted:** "Top up here: {topup_url}."
   - **TEAM member, low/critical:** "Try cheaper models like {cheapest_model} for routine tasks. Contact your team admin to increase the budget."
   - **TEAM member, exhausted:** "Contact your team admin to increase the budget."
   - **Voucher recipient, low/critical:** "Try cheaper models like {cheapest_model} for routine tasks. Run `request_topup` to ask the issuing admin for more budget."
   - **Voucher recipient, exhausted:** "Run `request_topup` to ask the issuing admin for more budget."
   - When `cheapest_model` is null: drop the "Try cheaper models like..." clause.
7. **Where the warning appears:**
   - `my_status` response — `warning?: Warning | null` field at top level
   - `my_budget` response — same `warning?` field
   - `my_recent_usage` response — same `warning?` field
   - `BudgetSnapshot` (the canonical type) gains the `warning?` field — additive, doesn't break callers
8. **Currency-aware:** the formatted amounts in messages use M3a's `formatCurrency` with the org's currency and default locale per currency.
9. **Dashboard surfacing:** if `BudgetSnapshot.warning` is present in any API response the dashboard consumes, render `<BudgetWarningBanner/>` at the top of `/dashboard/overview`, `/usage`, `/bundles`:
   - `low` → yellow banner
   - `critical` → orange banner
   - `exhausted` → red banner
   - Banner shows `message` + `suggestion.text`. CTA button rules:
     - If `topup_url` is non-null → "Top up" button linking to that URL
     - If `topup_via_mcp_tool` is non-null → display "Run `request_topup` from your AI client" as inline text (no button — it's an MCP-side action)
     - Otherwise → "Contact admin" plain text (no button)
10. **Dashboard banner visibility:** TEAM admins and TEAM members both see the banner with branched wording. Voucher recipients are synthetic users who rarely visit the dashboard; they see the warning in MCP tool responses (the universal surface). The banner is purely a TEAM-side addition; no special voucher-banner work required.
11. **Webhooks unaffected.** No `warning` field in webhook payloads.
12. **Tests deterministic:** `remaining_pct` is rounded to nearest integer for stability.
