# Replit Prompt — Hotfix: voucher-aware OAuth credential form rejects all real browser submissions

## What & Why

The voucher-aware OAuth credential form (Task #51) is **production-broken** for voucher recipients and API-key holders coming from any third-party MCP host (Claude.ai, ChatGPT, Gemini).

**Symptom:** any real browser submission of the credential form returns `400 invalid_request: oauth_continue must be a /oauth/authorize URL on this origin`.

**Root cause:** the `isSafeContinue` validator in `server/lib/oauth/authorize-credential.ts:71-80` blanket-rejects any `oauth_continue` containing `://`, even when it's safely inside a query-string parameter:

```ts
if (value.includes("://")) return false;
```

But `authorize.ts` renders the form with the original /oauth/authorize URL embedded as `oauth_continue`, and that URL always contains `redirect_uri=https://...` (the OAuth client's callback, e.g. `https://claude.ai/callback`). When a real browser submits the form, the value DOM-decodes (`&amp;` → `&`) and the server URL-decodes the form body, ending up with `://` in the parsed string. Validator rejects it.

**Why no live user has hit this yet:** every "successful" OAuth flow verified in QA went through with an already-logged-in Allotly session cookie. When `req.session.userId` is set, `authorize.ts` skips the credential form rendering entirely and proceeds to consent. The credential form path has never been exercised end-to-end in production.

**Why the existing 12-case test suite in `tests/oauth/credential-paths.test.ts` didn't catch this:** the tests call `authorizeCredentialHandler` directly with a JS object via `mockReq()`. They never round-trip through actual form submission, so the test's `oauth_continue` value is whatever JS string the test sets — never the value a real browser would submit (which contains `://` in `redirect_uri=`).

## Done looks like

- `oauth_continue` values shaped exactly like what the rendered form embeds (`/oauth/authorize?client_id=...&redirect_uri=https://claude.ai/cb&code_challenge=...&...`) are accepted by the validator.
- Open-redirect protections previously enforced still hold: `https://evil.example.com/oauth/authorize`, `//evil.com/oauth/authorize`, `\\evil.com\oauth\authorize`, `/oauth/authorize/../foo` are all still rejected.
- A new regression test exercises the actual production code path: it renders the credential form via `authorizeHandler`, extracts the embedded `oauth_continue` value verbatim, and POSTs it to `authorizeCredentialHandler`. Without the fix this test fails with the 400 we're chasing; with the fix it succeeds.
- All existing `tests/oauth/credential-paths.test.ts` cases still pass unchanged.

## Steps

### 1. Replace `isSafeContinue` in `server/lib/oauth/authorize-credential.ts`

Locked replacement:

```ts
/**
 * `oauth_continue` MUST be a relative path on our origin pointing back at
 * /oauth/authorize. This blocks open-redirect via crafted hidden field while
 * allowing the legitimate form-rendered value (which includes nested
 * `redirect_uri=https://...` inside the query string).
 *
 * Implementation: parse the value relative to a placeholder origin. If the
 * parser interprets the value as same-origin (hostname unchanged) AND the
 * pathname is exactly /oauth/authorize, it is safe. Anything else (absolute
 * URL with a real host, protocol-relative `//evil.com/...`, traversal, etc.)
 * either changes the hostname or moves the pathname off /oauth/authorize.
 *
 * The `\\` check stays because some browser/Node URL parsers disagree on
 * whether backslash is a path separator; defensively reject it before parse.
 */
function isSafeContinue(value: unknown): value is string {
  if (typeof value !== "string") return false;
  if (value.startsWith("//")) return false;
  if (value.includes("\\")) return false;
  try {
    const url = new URL(value, "http://placeholder.local");
    if (url.hostname !== "placeholder.local") return false;
    if (url.pathname !== "/oauth/authorize") return false;
    return true;
  } catch {
    return false;
  }
}
```

Do not touch any other function in this file. Do not change the call site of `isSafeContinue` — the signature is preserved.

### 2. Add a regression test in `tests/oauth/credential-paths.test.ts`

Add one new `it()` block at the end of the existing `describe()`. The test must round-trip through the actual credential-form-render → form-submission code path, not bypass it:

```ts
it("[regression] accepts the oauth_continue value the credential form actually embeds", async () => {
  // GET /oauth/authorize without a session → renders the credential form,
  // which embeds the original URL as <input name="oauth_continue" value="...">
  const aReq = mockReq({
    query: {
      client_id: registeredClientId,
      redirect_uri: TEST_REDIRECT_URI,                    // contains http://localhost:4444/cb → ://
      response_type: "code",
      code_challenge: pkceChallenge,
      code_challenge_method: "S256",
      scope: "mcp",
      state: "embedded-continue-regression",
      resource: MCP_AUDIENCE,
    },
    session: {},
    originalUrl: "/oauth/authorize?client_id=" + registeredClientId
      + "&redirect_uri=" + encodeURIComponent(TEST_REDIRECT_URI)
      + "&response_type=code&code_challenge=" + pkceChallenge
      + "&code_challenge_method=S256&scope=mcp&state=embedded-continue-regression"
      + "&resource=" + encodeURIComponent(MCP_AUDIENCE),
  });
  const aRes = mockRes();
  await authorizeHandler(aReq as any, aRes as any);
  expect(aRes.statusCode).toBe(200);

  // Extract the oauth_continue value EXACTLY as the form embeds it, then
  // simulate the browser's DOM-decode of HTML entities (&amp; → &). This is
  // what `req.body.oauth_continue` looks like server-side after a real form
  // submission.
  const html = String(aRes.body);
  const match = html.match(/name="oauth_continue" value="([^"]*)"/);
  expect(match).not.toBeNull();
  const embedded = match![1].replace(/&amp;/g, "&");

  // Sanity: the value MUST contain `://` (it has redirect_uri=http(s)://...).
  // If this assertion ever fails the test no longer guards against the bug.
  expect(embedded).toMatch(/:\/\//);

  // Extract the CSRF the form was rendered with.
  const csrfMatch = html.match(/name="csrf" value="([^"]*)"/);
  expect(csrfMatch).not.toBeNull();
  const csrfFromForm = csrfMatch![1];

  // Carry the same session through to the POST so CSRF checks pass.
  const sessionForPost: any = aReq.session;

  // POST credential with a deliberately malformed voucher code so we don't
  // mint a real user. We're proving validation passes the oauth_continue
  // check; downstream credential validation can fail freely. The 400
  // "oauth_continue must be a /oauth/authorize URL" must NOT appear; the
  // 401 form re-render with the credential-error banner is the success
  // signal here.
  const cReq = mockReq({
    body: {
      csrf: csrfFromForm,
      oauth_continue: embedded,
      credential_type: "voucher",
      code: "ALLOT-FAKE-FAKE-FAKE",
    },
    session: sessionForPost,
  });
  const cRes = mockRes();
  await authorizeCredentialHandler(cReq as any, cRes as any);

  // Acceptance gate: must NOT be the 400 oauth_continue rejection. Anything
  // downstream (401 with form re-render, or generic-error response) proves
  // the validator no longer drops legitimate browser submissions.
  expect(cRes.statusCode).not.toBe(400);
  // Concretely: with a malformed voucher, expect 401 + form re-render.
  expect(cRes.statusCode).toBe(401);
  expect(String(cRes.body)).toMatch(/data-testid="credential-error"/);
});
```

Place the test inside the existing `describe(...)` block at the end. Reuse the existing `mockReq`, `mockRes`, `authorizeHandler`, `authorizeCredentialHandler`, `registeredClientId`, `TEST_REDIRECT_URI`, `pkceChallenge`, and `MCP_AUDIENCE` fixtures.

it("[regression] isSafeContinue still rejects the previously-rejected attack shapes", async () => {
  const sessionWithCsrf: any = { _oauthCsrf: "stable-csrf-token" };

## Architectural constraints

- The `\\` reject-before-parse check stays — some browser/Node URL parsers disagree on backslash interpretation, so the defensive reject runs first.
- `new URL(value, "http://placeholder.local")` is the standard cross-runtime approach for "is this a same-origin path or something else." `placeholder.local` is an arbitrary RFC 6761 reserved TLD that no real host can use.
- Do not relax the `pathname === "/oauth/authorize"` check (allowing `/oauth/authorize/anything` would re-introduce a path-confusion attack surface).
- Do not allow the value to start with `//` (protocol-relative) — `new URL` resolves `//evil.com/x` to `http://evil.com/x` against most bases, but defense in depth.
