Allotly — Azure OpenAI Provider: Replit Build Prompts
Context for the Replit Agent
Allotly is a SaaS platform ("The AI Spend Control Plane") with a unified thin proxy at allotly.ai/api/v1. All requests from all users — Teams and Vouchers — flow through this proxy. The proxy currently supports three providers: OpenAI, Anthropic, and Google Gemini.
A customer needs to use Azure OpenAI (they call it "Nebula One") instead of direct OpenAI. Azure OpenAI runs OpenAI's models on Azure infrastructure but uses a different endpoint format, a different auth header, and deployment names instead of model IDs.
This work adds Azure OpenAI as a named 4th provider — not a generic endpoint. This means dedicated adapter, dedicated validation, dedicated model catalogue entries with verified pricing, and a proper connection UI card.
Key Azure OpenAI differences from direct OpenAI

Two endpoint formats exist (we must support both):
a) Legacy versioned API (pre-August 2025, still widely used):

   POST https://{base-url}/openai/deployments/{deployment-name}/chat/completions?api-version=2024-10-21

Deployment name is in the URL path, NOT in the request body
api-version query parameter is required
model field in body is optional/ignored (deployment in URL takes precedence)

b) New v1 API (August 2025+, recommended going forward):
   POST https://{base-url}/openai/v1/chat/completions

No api-version query parameter needed
model field in the request body IS required (set to the deployment name)
Much more OpenAI-compatible — nearly a drop-in swap


Two hostname formats exist (we must accept both):

{resource}.openai.azure.com — the original Azure OpenAI hostname
{resource}.services.ai.azure.com — the newer Microsoft Foundry hostname
Both work identically. The admin may have either depending on when they set up their Azure resource.


Auth header: api-key: {key} (NOT Authorization: Bearer {key})
Model naming: Azure uses admin-defined deployment names (e.g. my-gpt4o) that map to underlying OpenAI models (e.g. gpt-4o).
Request/response body: Identical to OpenAI format. Same messages, max_tokens, temperature, etc. Same response shape with choices, usage, etc.
Streaming: Identical SSE format to OpenAI.

The ProviderConnection model currently stores: provider (enum), apiKeyEncrypted, apiKeyIv, displayName, status, lastValidatedAt, and the org/team relations. Add four new nullable fields for Azure-specific configuration:

prismamodel ProviderConnection {
  // ... existing fields ...
  
  azureBaseUrl         String?   @map("azure_base_url")          // Full base URL, e.g. "https://contoso.openai.azure.com" or "https://contoso.services.ai.azure.com"
  azureApiVersion      String?   @map("azure_api_version")       // e.g. "2024-10-21". Only used in legacy mode. Null = use v1 mode.
  azureEndpointMode    String?   @map("azure_endpoint_mode")     // "v1" or "legacy". Defaults to "v1" if null.
  azureDeployments     Json?     @map("azure_deployments")       // JSON: array of {deploymentName, modelId, inputPricePer1M, outputPricePer1M}
}
Why azureBaseUrl instead of azureResourceName: Azure endpoints come in two hostname formats — {resource}.openai.azure.com and the newer {resource}.services.ai.azure.com. Rather than trying to parse and reconstruct these, we store the full base URL exactly as the admin provides it. This is future-proof if Microsoft introduces more hostname variants.
Why azureEndpointMode: The v1 API and the legacy versioned API construct URLs differently and handle the model field in the body differently. Rather than guessing, we let the admin choose (defaulting to v1 which is the recommended path going forward). In legacy mode, azureApiVersion is required.
The azureDeployments JSON field stores the admin's deployment-to-model mapping. Example value:
json[
  { "deploymentName": "my-gpt4o", "modelId": "gpt-4o", "inputPricePer1M": 250, "outputPricePer1M": 1000 },
  { "deploymentName": "my-gpt4o-mini", "modelId": "gpt-4o-mini", "inputPricePer1M": 15, "outputPricePer1M": 60 },
  { "deploymentName": "nebula-one", "modelId": "gpt-4o", "inputPricePer1M": 250, "outputPricePer1M": 1000 }
]
These fields are only used when provider === AZURE_OPENAI. They remain null for other providers.

A2. Provider Detection in Proxy
File: server/lib/proxy/translate.ts — function detectProvider()
The current routing logic uses model name prefixes:

gpt-*, o1*, o3*, o4* → OPENAI
claude-* → ANTHROPIC
gemini-* → GOOGLE

Azure OpenAI is different: the model name in the request body is the admin's deployment name (e.g. nebula-one), which doesn't have a recognisable prefix. So we cannot use prefix-based routing for Azure.
New routing logic:

First, check prefix-based routing for the three existing providers (unchanged).
If no prefix matches, check whether the model name matches any deploymentName in any active AZURE_OPENAI provider connection's azureDeployments array for the requesting user's organisation.
If a match is found, return AZURE_OPENAI as the provider.
If no match is found anywhere, return the existing error ("Model not supported").

Implementation approach:
The deployment-name lookup needs to be fast (it runs on every request). Cache the org's Azure deployment mappings in Redis:
Key:    allotly:azure-deployments:{orgId}
Value:  JSON string of the azureDeployments array from the ProviderConnection
TTL:    300 seconds (5 minutes)
Invalidate this cache key whenever the admin updates their Azure provider connection or deployment mappings.
The detectProvider() function signature needs to change. It currently only takes the model name. It now also needs the orgId (or the full membership context) so it can look up Azure deployment mappings. Update the function signature:
typescript// OLD:
function detectProvider(model: string): Provider

// NEW:
function detectProvider(model: string, orgId: string): Promise<{ provider: Provider; azureDeployment?: AzureDeploymentMapping }>
Where AzureDeploymentMapping is:
typescriptinterface AzureDeploymentMapping {
  deploymentName: string;    // What the user sends as "model"
  modelId: string;           // The underlying OpenAI model (for reference/logging)
  inputPricePer1M: number;   // Cents per 1M input tokens
  outputPricePer1M: number;  // Cents per 1M output tokens
}
Since this function is now async (Redis lookup), update all call sites in handler.ts to await it.
A3. Provider Adapter — Azure OpenAI
New functions to add in server/lib/proxy/translate.ts:
validateAzureOpenAIKey()
Validates the admin's Azure OpenAI API key by making a lightweight request. The simplest validation is to call the deployments list endpoint (if available) or to call the chat completions endpoint with a minimal payload against the first configured deployment.
typescriptasync function validateAzureOpenAIKey(
  apiKey: string, 
  baseUrl: string,           // e.g. "https://contoso.openai.azure.com"
  endpointMode: "v1" | "legacy",
  apiVersion: string | null, // Only used in legacy mode
  deployments: AzureDeploymentMapping[]
): Promise<{ valid: boolean; error?: string }> {
  // Use the first deployment to make a minimal test call
  const deployment = deployments[0];
  if (!deployment) return { valid: false, error: "No deployments configured" };
  
  // Construct URL based on endpoint mode
  let url: string;
  let body: any;
  
  if (endpointMode === "v1") {
    // v1 mode: model in body, no api-version, simpler URL
    url = `${baseUrl}/openai/v1/chat/completions`;
    body = {
      model: deployment.deploymentName,
      messages: [{ role: "user", content: "test" }],
      max_tokens: 1,
    };
  } else {
    // Legacy mode: deployment in URL path, api-version required
    const version = apiVersion || "2024-10-21";
    url = `${baseUrl}/openai/deployments/${deployment.deploymentName}/chat/completions?api-version=${version}`;
    body = {
      messages: [{ role: "user", content: "test" }],
      max_tokens: 1,
    };
  }
  
  try {
    const response = await fetch(url, {
      method: "POST",
      headers: {
        "Content-Type": "application/json",
        "api-key": apiKey,
      },
      body: JSON.stringify(body),
    });
    
    if (response.status === 401 || response.status === 403) {
      return { valid: false, error: "Invalid API key" };
    }
    if (response.status === 404) {
      return { valid: false, error: `Deployment '${deployment.deploymentName}' not found. Check the deployment name and base URL.` };
    }
    // 200 or 429 (rate limited but key is valid) both indicate a valid key
    return { valid: true };
  } catch (err) {
    return { valid: false, error: `Connection failed: ${err.message}. Check that the base URL is correct.` };
  }
}
translateRequestForAzureOpenAI()
Azure OpenAI accepts the same request body as OpenAI but:

The model field in the body is optional (the deployment name is in the URL path)
The endpoint URL is different
Reasoning models still need max_completion_tokens instead of max_tokens

typescriptfunction translateRequestForAzureOpenAI(
  body: any, 
  deployment: AzureDeploymentMapping,
  baseUrl: string,            // e.g. "https://contoso.openai.azure.com"
  endpointMode: "v1" | "legacy",
  apiVersion: string | null,  // Only used in legacy mode
  stream: boolean
): { url: string; translatedBody: any } {
  
  // Clone body
  const translatedBody = { ...body };
  
  // Construct URL and handle model field based on endpoint mode
  let url: string;
  
  if (endpointMode === "v1") {
    // v1 mode: simple URL, model field stays in body (set to deployment name)
    url = `${baseUrl}/openai/v1/chat/completions`;
    translatedBody.model = deployment.deploymentName;
  } else {
    // Legacy mode: deployment in URL path, api-version required, model removed from body
    const version = apiVersion || "2024-10-21";
    url = `${baseUrl}/openai/deployments/${deployment.deploymentName}/chat/completions?api-version=${version}`;
    delete translatedBody.model;  // Deployment is in the URL, not the body
  }
  
  // Handle reasoning models (o1, o3, o4 series) — same logic as OpenAI
  const underlyingModel = deployment.modelId.toLowerCase();
  if (underlyingModel.startsWith("o1") || underlyingModel.startsWith("o3") || underlyingModel.startsWith("o4")) {
    if (translatedBody.max_tokens && !translatedBody.max_completion_tokens) {
      translatedBody.max_completion_tokens = translatedBody.max_tokens;
      delete translatedBody.max_tokens;
    }
  }
  
  // NOTE: Auth headers are NOT set here. They are injected at step 18 (setProviderAuth)
  // in handler.ts, which adds the "api-key" header using the decrypted provider key.
  
  return { url, translatedBody };
}
